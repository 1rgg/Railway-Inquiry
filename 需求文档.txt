静态火车票查询网页需求文档
一、项目概述
1.1 项目背景
基于已提供的 12306 车次查询开放 API，开发一款纯前端静态火车票查询网页，无需后端服务，部署后可直接在浏览器端运行，满足用户快速查询车站车次、站间直达车次及中转方案的需求。
1.2 项目目标
实现车站维度的车次信息查询，完整还原 API 返回的所有车次字段
支持站间直达车次筛选、中转方案智能匹配两种核心出行查询场景
界面遵循简约设计原则，信息层级清晰，操作路径短，适配桌面端与移动端
纯静态实现，无服务端依赖，可直接部署在任意静态托管平台
保障页面加载、交互、渲染全流程性能达标，提供流畅的使用体验
二、功能需求
2.1 单站车次查询（API 原生能力）
用户输入单个车站 / 城市名称，查询该车站所有出发车次信息，是其他查询功能的基础能力。
输入项：车站名称（文本输入框）
触发方式：点击「查询」按钮或回车触发
结果展示：
顶部展示查询车站名、总车次数量、总页数
列表展示每趟车次的：车次号、始发站、终到站、出发时间、候车室 / 检票口、运行状态
支持分页切换，每页默认 10 条，与 API 分页能力对齐
补充：支持页码跳转，默认展示第 1 页
2.2 直达车次查询
用户输入出发站与到达站，筛选出两站之间的直达车次。
输入项：出发站、到达站（两个文本输入框）
实现逻辑：前端调用出发站的单站车次查询接口，在返回的车次列表中，筛选「目的地」字段与到达站完全匹配的车次
结果展示：
展示符合条件的直达车次总数
车次列表字段与单站查询保持一致
无匹配结果时，展示「暂无直达车次，可尝试中转方案」提示，并引导跳转至中转查询
2.3 中转方案查询
针对无直达车次的场景，提供中转出行方案查询能力。
输入项：出发站、到达站、中转城市（支持手动输入，同时提供热门中转城市快捷选择，如北京、上海、广州、郑州、武汉等）
实现逻辑：
分别调用 API 查询「出发站→中转城市」的直达车次、「中转城市→到达站」的直达车次
前端进行车次组合匹配：中转到达时间需早于下一段车次的出发时间，且默认保留中转停留 1 小时以上的合理方案
按总行程时长升序排序，优先展示耗时最短的方案
结果展示：
每个中转方案为独立卡片，包含：总行程耗时、中转停留时长
分段展示两段行程的车次号、出发 / 到达站、出发 / 到达时间、检票口、状态
标注中转站点名称，突出中转停留时间
2.4 通用交互功能
功能切换：页面顶部设置选项卡，支持「单站查询」「直达查询」「中转查询」一键切换
输入校验：查询前校验必填项是否为空，为空时给出轻量提示
加载状态：API 请求过程中展示加载动画，禁止重复提交
错误处理：请求失败、接口异常时，展示友好的错误提示
本地记忆：可选保存用户最近 5 次查询记录（存储在浏览器本地），点击可快速复现查询
三、界面设计要求
3.1 整体风格
简约扁平化设计，去除冗余装饰元素，以信息展示为核心
主色调采用铁路经典深蓝色系（参考值：#165DFF），搭配浅灰色背景，文字以深灰、黑色为主，保证可读性
圆角、弱阴影设计，视觉风格柔和清爽
3.2 页面布局
页面采用上下结构，整体居中对齐，最大宽度限制在 1200px：
顶部：页面标题「火车票查询」，简洁醒目
中部：查询操作区，包含功能选项卡、输入框组、查询按钮
下部：结果展示区，无查询时展示默认占位提示，有结果时展示列表 / 卡片
底部：极简分页控件（仅单站、直达查询展示）
3.3 响应式适配
桌面端：查询区横向排列，结果列表以表格形式展示
移动端（宽度≤768px）：查询区纵向堆叠，结果列表转为卡片式布局，保证单指可操作
字体、间距随屏幕尺寸自适应，避免横向滚动
四、API 对接说明
4.1 接口基础信息
请求地址：https://tmini.net/api/12306
请求方式：GET
返回格式：JSON
跨域说明：前端直接通过 fetch/axios 发起请求，需确认接口支持 CORS；若存在跨域问题，需在需求中备注可通过静态代理或 JSONP 方式兼容
4.2 请求参数
所有查询场景均复用该接口，按需传参：
表格
参数名	必填	类型	说明	传参规则
msg	是	string	需要查询的城市 / 车站名称	对应用户输入的车站名
ckey	否	string	后台 KEY	默认不传，保留扩展能力
page	否	string	页码	单站 / 直达查询分页时传入，默认 1
encoding	否	string	返回格式	固定传 json
注意：官方请求示例存在拼写错误（encoding=josn），实际调用时需修正为 encoding=json。
4.3 返回参数说明
前端需完整解析以下字段并用于渲染：
根级字段：车站、当前页码、总页数、总车次、每页条数、当前页车次列表
车次列表子字段：车次号、出发地、目的地、出发时间、候车室 / 检票口、状态
特殊处理：「候车室 / 检票口」字段为 -- 时，前端展示为「暂无信息」
4.4 调用示例
plaintext
https://tmini.net/api/12306?msg=长春&encoding=json&page=1

五、技术实现要求
5.1 技术栈
结构：HTML5 语义化标签
样式：原生 CSS3，优先使用 Flex/Grid 布局，可引入 Tailwind CSS 简化开发（可选）
交互：原生 JavaScript（ES6+），不强制引入 jQuery 等重型库
无后端依赖，所有逻辑均在浏览器端完成
5.2 代码规范
代码结构分层清晰：HTML 结构、CSS 样式、JS 逻辑分离
关键逻辑添加注释，尤其是中转匹配算法、分页逻辑
变量命名语义化，避免无意义缩写
兼容主流现代浏览器（Chrome、Edge、Firefox、Safari），无需兼容 IE 系列
六、网页性能要求
6.1 核心加载性能指标
首屏渲染时间：WiFi 环境下首屏完整内容渲染耗时≤1s，4G 网络环境下≤1.5s，弱网（3G）环境下≤3s
核心 Web Vitals 指标达标：
最大内容绘制（LCP）≤2.5s
首次输入延迟（FID）≤100ms
累积布局偏移（CLS）≤0.1
页面完全加载后，空闲状态下浏览器内存占用≤50MB，无内存泄漏问题
6.2 静态资源优化要求
整体静态资源（HTML+CSS+JS）原始总体积≤200KB；开启 gzip/br 压缩后总体积≤50KB
禁止引入不必要的第三方库与重型框架，若使用工具类库需按需引入，禁止全量加载冗余代码
优先使用系统默认字体，不引入外部字体文件；基础图标采用 CSS 绘制或内联 SVG，不调用大容量图标库
CSS 样式单文件引入，避免多层级嵌套与冗余选择器；JS 代码去除无用逻辑、注释与死代码，精简体积
6.3 渲染与交互性能
单页 10 条车次列表渲染耗时≤100ms，分页切换、Tab 功能切换无白屏、无明显卡顿
中转方案计算：单次匹配 100 条以内车次数据时，计算耗时≤300ms；计算过程不阻塞主线程，配合加载提示反馈状态
页面滚动帧率稳定在 60fps，列表滚动、页面滑动无掉帧、粘滞现象
按钮点击、输入框输入、选项切换等基础交互的响应延迟≤50ms，无操作滞后感
采用批量 DOM 更新机制，避免频繁触发重排重绘，减少渲染性能损耗
6.4 网络请求优化
查询操作添加防抖处理，防抖时长 300ms，避免用户连续输入 / 点击产生无效重复请求
对相同车站、相同页码的查询结果做本地短期缓存（有效期 5 分钟），重复查询时优先读取本地缓存，降低网络请求频次
中转查询的两段 API 请求采用并行发起模式，缩短整体查询等待时长
请求失败自动重试 1 次，重试失败后再展示错误提示，提升弱网环境下的可用性
合理设置请求超时时间（默认 10s），超时后给出明确提示，避免页面长时间无响应
6.5 移动端性能
移动端 4G 环境下页面首屏加载完成时间≤2s
触摸点击、滑动等交互响应延迟≤100ms，符合移动端操作体验标准
禁止使用高频定时器与密集型动画，降低移动端电量消耗
列表滑动保持跟手流畅，无卡顿、掉帧现象
七、异常与边界处理
输入为空：点击查询时提示「请输入车站名称」，不发起接口请求
无查询结果：接口返回车次列表为空时，展示「暂无符合条件的车次」占位图与文案
接口请求失败：提示「网络异常，请稍后重试」，保留用户输入内容
中转无方案：两段行程无法匹配出合理中转时，提示「未找到合适的中转方案，可尝试更换中转城市」
分页边界：第一页时禁用上一页，最后一页时禁用下一页
八、交付要求
交付单个 HTML 文件或完整静态资源包（html+css+js 文件夹），双击即可运行
页面默认展示直达查询入口，功能切换流畅无卡顿
附带简单的使用说明，标注接口参数与功能逻辑
界面还原简约设计要求，无多余广告、冗余元素
核心性能指标达标，无明显加载慢、交互卡顿问题
